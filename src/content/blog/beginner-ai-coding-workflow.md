AI 코딩 도구를 처음 쓰면 “할 일 앱 만들어 줘”처럼 결과 전체를 요청하기 쉽습니다. 코드는 빠르게 생기지만, 오류가 났을 때 어느 부분을 고쳐야 하는지 알기 어렵고 기존 기능이 조용히 망가질 수도 있습니다.

이 글에서는 VS Code와 GitHub Copilot Chat을 사용해 기존 JavaScript 할 일 앱에 **완료 항목 숨기기** 기능을 추가한다고 가정합니다. 중요한 것은 특정 프레임워크 문법이 아니라 다음 루프입니다.

> 계획을 확인한다 → 작은 변경만 맡긴다 → 직접 실행한다 → 테스트와 diff를 검토한다

![계획, 작은 코드 변경, 테스트, diff 검토가 반복되는 AI 코딩 흐름](/images/posts/ai-coding-workflow.jpg)

## 시작 전에 안전장치 세 개를 만든다

AI에게 저장소를 맡기기 전에 현재 상태를 고정합니다.

```bash
git status
npm test
git switch -c feature/hide-completed
```

`git status`에 내가 만든 미커밋 변경이 있다면 먼저 무엇인지 확인합니다. `npm test`는 기능을 고치기 전에도 테스트가 통과하는지 보는 기준점입니다. 새 브랜치는 변경을 되돌리거나 비교하기 쉽게 해 줍니다.

프로젝트에 테스트가 없다면 최소한 현재 화면과 동작을 메모합니다.

- 새 할 일을 추가할 수 있다.
- 체크박스를 누르면 완료 상태가 바뀐다.
- 새로고침 후 상태가 유지된다.
- 빈 문자열은 추가되지 않는다.

이 네 줄이 나중의 수동 테스트 목록이 됩니다.

## 1단계: 코드를 고치기 전에 지도를 요청한다

Copilot Chat에 구현을 바로 맡기지 않습니다. 먼저 관련 파일과 데이터 흐름을 찾게 합니다.

```text
이 저장소는 처음 보는 초보 개발자야.
‘완료 항목 숨기기’ 기능을 추가하려고 해.

아직 파일을 수정하지 말고 다음을 조사해 줘.
1. 할 일 목록을 화면에 그리는 파일과 함수
2. 완료 상태를 변경하는 코드
3. 필터나 로컬 저장소 관련 코드
4. 관련 테스트 파일

각 파일이 맡는 역할과 가장 작은 구현 순서를 설명해 줘.
추측한 부분은 ‘추측’이라고 표시해 줘.
```

답변에서 파일명을 눌러 실제 코드를 엽니다. 존재하지 않는 함수나 파일을 말한다면 그 상태로 구현을 진행하지 않습니다. AI가 저장소 전체를 정확히 읽었다고 가정하는 대신, **지도와 실제 파일이 일치하는지** 먼저 확인합니다.

## 2단계: 요구사항을 테스트 가능한 문장으로 바꾼다

“완료 항목을 숨겨 줘”에는 결정되지 않은 부분이 많습니다. 다음처럼 동작을 고정합니다.

```text
요구사항:
- 목록 위에 ‘완료 항목 숨기기’ 체크박스를 추가한다.
- 체크하면 completed가 true인 항목만 화면에서 숨긴다.
- 체크를 풀면 기존 순서 그대로 다시 보인다.
- 할 일 데이터 자체를 삭제하지 않는다.
- 새로고침하면 체크박스는 기본값인 해제 상태로 돌아온다.
- 기존 추가/완료/삭제 동작은 유지한다.
```

`데이터를 삭제하지 않는다`, `새로고침 후 기본값` 같은 조건이 없으면 AI가 임의로 결정합니다. 원하는 동작을 나중에 고치는 것보다 처음부터 관찰 가능한 문장으로 적는 편이 빠릅니다.

## 3단계: 구현보다 테스트를 먼저 제안받는다

테스트 환경이 있다면 구현 전에 테스트 사례를 요청합니다.

```text
현재 테스트 도구와 기존 테스트 스타일을 먼저 확인해 줘.
코드는 수정하지 말고 이 기능에 필요한 테스트 사례만 제안해 줘.
정상 동작, 경계 조건, 기존 기능 회귀를 나눠서 작성해 줘.
```

좋은 제안에는 최소한 아래가 포함되어야 합니다.

1. 필터가 꺼져 있으면 완료/미완료 항목이 모두 보인다.
2. 필터를 켜면 완료 항목만 사라진다.
3. 필터를 다시 끄면 항목 순서가 유지된다.
4. 숨겨진 항목의 데이터는 배열에 남아 있다.
5. 필터가 켜진 상태에서도 새 미완료 항목을 추가할 수 있다.

테스트를 먼저 생각하면 “보이지만 데이터에서 삭제된” 잘못된 구현을 초기에 잡을 수 있습니다.

## 4단계: 한 번에 한 층만 수정한다

이제 구현을 두 번으로 나눕니다.

### 첫 번째 요청: 상태와 필터 함수

```text
첫 단계만 구현해 줘.

- hideCompleted 상태를 추가한다.
- 원본 todos 배열을 바꾸지 않고 화면에 표시할 배열을 계산한다.
- 기존 코드 스타일과 이름 규칙을 따른다.
- UI는 아직 추가하지 않는다.
- 관련 단위 테스트를 실행하고 결과를 알려 줘.
```

예상되는 핵심 로직은 복잡하지 않습니다.

```js
function getVisibleTodos(todos, hideCompleted) {
  if (!hideCompleted) return todos;
  return todos.filter((todo) => !todo.completed);
}
```

여기서 확인할 점은 `filter`의 문법보다 원본 배열을 덮어쓰지 않는다는 사실입니다. 원본을 삭제해 버리면 체크를 풀었을 때 완료 항목을 복구할 수 없습니다.

### 두 번째 요청: UI 연결

```text
이제 두 번째 단계만 구현해 줘.

- 목록 위에 체크박스와 label을 추가한다.
- 체크 상태를 hideCompleted와 연결한다.
- 키보드와 스크린 리더에서도 이름을 알 수 있게 한다.
- 기존 CSS 구조를 최대한 유지한다.
- 전체 테스트를 실행한다.
```

작업을 나누면 오류가 상태 로직에 있는지, UI 이벤트 연결에 있는지 범위를 빠르게 좁힐 수 있습니다.

## 5단계: AI의 “완료” 대신 직접 네 가지를 본다

AI가 테스트가 통과했다고 말해도 터미널에서 직접 실행합니다.

```bash
npm test
npm run lint
npm run build
git diff --check
```

프로젝트에 없는 명령은 억지로 실행하지 말고 `package.json`의 `scripts`를 확인합니다. 각 명령의 목적은 다릅니다.

- `test`: 기대한 동작이 유지되는가
- `lint`: 문법·스타일·일부 오류 패턴이 없는가
- `build`: 배포 가능한 결과가 만들어지는가
- `git diff --check`: 공백 오류나 충돌 흔적이 없는가

그다음 앱을 열어 체크박스를 키보드로 조작하고, 추가·완료·삭제·새로고침을 직접 시험합니다. 자동 테스트가 화면의 레이블 위치나 사용성을 항상 잡아 주는 것은 아닙니다.

## 6단계: diff를 ‘설명 시험지’처럼 읽는다

```bash
git diff --stat
git diff
```

먼저 `--stat`으로 예상보다 많은 파일이 바뀌지 않았는지 봅니다. 체크박스 하나를 추가했는데 설정 파일 열 개와 의존성 잠금 파일이 바뀌었다면 이유를 물어야 합니다.

각 변경 줄에 대해 스스로 답해 봅니다.

- 이 줄이 요구사항 중 어느 항목을 만족시키는가
- 이 줄을 지우면 어떤 테스트가 실패해야 하는가
- 사용자 입력이나 외부 데이터가 들어오는 지점인가
- 더 단순한 기존 함수로 처리할 수 있었는가

이해되지 않는 부분은 Copilot에게 범위를 좁혀 묻습니다.

```text
현재 diff 중 getVisibleTodos 변경만 설명해 줘.
입력과 반환값을 실제 예시 배열로 보여 주고,
원본 배열이 바뀌지 않는 이유도 설명해 줘.
코드는 추가로 수정하지 마.
```

`코드는 수정하지 마`를 붙이면 설명을 듣는 중에 작업 범위가 다시 커지는 것을 막을 수 있습니다.

## 실패했을 때 쓰는 디버깅 프롬프트

오류 화면 캡처만 보내기보다 재현 정보와 기대 결과를 함께 줍니다.

```text
다음 테스트 실패 원인을 분석해 줘. 아직 수정하지 마.

[실행 명령]
npm test -- hide-completed

[기대 결과]
필터를 끄면 완료 항목이 원래 순서로 다시 나타난다.

[실제 결과]
완료 항목이 계속 보이지 않는다.

[오류 전문]
여기에 터미널 오류 전체 붙여넣기

[최근 변경]
git diff 내용 붙여넣기

가능성 높은 원인을 순서대로 적고, 각 원인을 확인할 최소 실험을 제안해 줘.
```

원인 분석과 수정을 분리하면 AI가 증상을 덮는 임시 코드를 바로 추가하는 일을 줄일 수 있습니다.

## 완료 기준

아래 항목을 모두 확인한 뒤 커밋합니다.

- 요구사항 여섯 줄을 실제 화면에서 확인했다.
- 기존 테스트와 새 테스트가 통과한다.
- lint와 build가 통과한다.
- diff에 설명할 수 없는 변경이 없다.
- API 키, 개인정보, 디버그 로그가 추가되지 않았다.
- 커밋 메시지가 변경 목적을 설명한다.

```bash
git add src tests
git commit -m "Add hide-completed todo filter"
```

GitHub의 공식 가이드도 Copilot이 잘하는 일로 테스트와 반복 코드를 들면서, 제안을 이해하고 자동 테스트·lint·코드 스캔 등으로 검증하라고 권합니다. AI 코딩의 핵심은 더 큰 코드를 한 번에 받는 것이 아니라, **작은 변경을 내가 검증할 수 있는 속도로 반복하는 것**입니다.
