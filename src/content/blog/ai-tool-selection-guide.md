AI 도구를 고를 때 기능 목록부터 보면 거의 모든 제품이 좋아 보입니다. “최신 모델”, “긴 문맥”, “에이전트” 같은 표현만으로는 내 일을 줄여 줄 도구인지 판단하기 어렵습니다.

이번에는 막 시작한 개발자 겸 일반 업무 사용자가 자주 만나는 세 가지 일을 기준으로 도구를 고르겠습니다.

1. 어제 발표된 기술 뉴스를 출처와 함께 조사한다.
2. 회사가 제공한 PDF 세 개에서 요구사항 차이를 찾는다.
3. 기존 저장소에서 작은 기능을 구현하고 테스트한다.

![검색, 문서 분석, 코딩 작업에 맞는 AI 도구를 고르는 작업대](/images/posts/ai-tool-selection.jpg)

## 결론표: 도구가 아니라 ‘입력 범위’를 고른다

| 작업 | 먼저 써볼 도구 | 강점 | 반드시 확인할 것 |
| --- | --- | --- | --- |
| 최신 웹 조사 | Perplexity | 웹을 검색하고 답변에 원문 링크를 붙인다 | 인용이 실제 주장을 뒷받침하는지 |
| 내 문서 비교 | NotebookLM | 선택한 출처를 중심으로 질문하고 인용 위치를 확인한다 | 업로드 문서의 버전과 누락된 부록 |
| 저장소 수정 | GitHub Copilot in VS Code | 코드 문맥에서 설명, 수정, 테스트 작성을 이어 간다 | diff, 테스트, 보안 문제 |

세 도구는 경쟁 제품처럼 보이지만 실제로는 **읽을 수 있는 입력의 범위**가 다릅니다. 웹 전체를 탐색해야 하는지, 내가 넣은 자료만 읽혀야 하는지, 현재 코드 저장소를 고쳐야 하는지를 먼저 정하면 선택이 쉬워집니다.

## 사례 1: “어제 발표된 정책 변경”을 조사한다

최신성이 중요한 질문은 웹 검색이 출발점입니다. Perplexity에 아래처럼 요청합니다.

```text
2026년 7월 14일 이후 공식 발표만 대상으로 [주제]의 변경 사항을 조사해 줘.

- 변경 전/후를 표로 비교
- 발표 기관의 원문 링크를 각 행에 표시
- 기사와 블로그는 보조 자료로만 사용
- 시행일과 발표일을 구분
- 확인되지 않은 해석은 별도 열에 표시
```

첫 답변에서 해야 할 일은 요약을 복사하는 것이 아니라 링크를 분류하는 것입니다.

- 공식 발표: 결정 근거로 사용
- 신뢰할 수 있는 해설: 영향 파악에 사용
- 출처가 불명확한 재인용: 제외

특히 날짜 질문에서는 “게시일”, “업데이트일”, “시행일”이 섞이기 쉽습니다. 최종 표에 날짜 열을 세 개로 나누면 잘못된 최신성 판단을 줄일 수 있습니다.

### 이 작업에서 NotebookLM이 첫 도구가 아닌 이유

NotebookLM은 내가 넣은 출처를 깊게 읽는 데 유리하지만, 조사 시작 시점에는 어떤 문서가 필요한지 아직 모릅니다. 먼저 웹에서 공식 문서 후보를 찾고, 문서가 모인 뒤 NotebookLM으로 넘기는 편이 효율적입니다.

## 사례 2: 계약서 PDF 세 개에서 요구사항 차이를 찾는다

이번에는 웹 지식보다 제공된 문서가 유일한 기준입니다. NotebookLM에 계약서, 제안요청서, 기술 부록을 각각 출처로 추가합니다. 파일 이름에는 버전과 날짜를 넣습니다.

```text
RFP_v3_2026-07-10.pdf
contract_draft_v2_2026-07-12.pdf
security_appendix_2026-07-08.pdf
```

그다음 모호한 “요약해 줘” 대신 비교 단위를 지정합니다.

```text
세 문서에서 다음 항목을 비교해 줘.

1. 데이터 보관 기간
2. 장애 응답 시간
3. 재위탁 조건
4. 계약 종료 후 삭제 기한

표의 각 셀에 문서명과 인용을 붙여 줘.
문서에 없는 내용은 추론하지 말고 ‘명시 없음’이라고 적어 줘.
서로 충돌하는 조항은 별도 목록으로 모아 줘.
```

예를 들어 RFP에는 “종료 후 30일 이내 삭제”, 계약서에는 “합리적인 기간 내 삭제”라고 되어 있다면 AI가 둘을 자연스럽게 합치지 않도록 해야 합니다. `충돌하는 조항`을 별도로 요구한 이유입니다.

### 실제 검수 방법

답변의 인용을 눌러 다음을 대조합니다.

- 조항 본문뿐 아니라 바로 앞의 정의 조항도 읽기
- `영업일`과 `일` 구분하기
- 본문과 부록 중 무엇이 우선하는지 확인하기
- 오래된 버전이 출처에 섞이지 않았는지 확인하기

민감한 회사 문서를 외부 서비스에 업로드할 때는 조직 정책과 해당 서비스의 데이터 처리 조건을 먼저 확인해야 합니다. 테스트는 공개 샘플 문서나 익명화한 사본으로 시작하는 편이 안전합니다.

## 사례 3: 기존 프로젝트에 ‘완료 항목 숨기기’를 추가한다

코드 저장소 안에서 파일을 읽고 수정해야 한다면 VS Code의 GitHub Copilot Chat을 씁니다. 바로 “기능 만들어 줘”라고 요청하지 않고 먼저 영향 범위를 묻습니다.

```text
이 저장소에서 할 일 목록을 렌더링하는 파일, 상태를 보관하는 파일,
관련 테스트 파일을 찾아 줘. 아직 수정하지 말고 다음만 알려 줘.

- 변경 후보 파일과 역할
- 현재 데이터 흐름
- 가장 작은 구현 순서
- 깨질 가능성이 있는 기존 동작
```

계획을 확인한 뒤 한 번에 한 단계만 요청합니다.

```text
첫 단계만 구현해 줘.
‘완료 항목 숨기기’ 상태를 추가하되 기존 필터 동작은 바꾸지 마.
수정 후 관련 테스트를 실행하고, 변경한 파일과 테스트 결과를 요약해 줘.
```

GitHub의 공식 권장사항도 복잡한 작업을 쪼개고 요구사항과 예시를 구체적으로 제공하며, 생성된 코드를 테스트와 자동화 도구로 검증하라고 안내합니다.

## 30분 도입 테스트: 느낌 대신 숫자로 고른다

후보 도구마다 같은 작업을 한 번씩 실행하고 아래 표를 기록합니다.

| 측정 항목 | 기록 방법 |
| --- | --- |
| 첫 결과 시간 | 질문 입력부터 검토 가능한 결과까지 분 |
| 근거 확인 시간 | 링크·인용·diff를 확인하는 데 든 분 |
| 수정 횟수 | 원하는 형식이 나오기까지 추가 요청 횟수 |
| 치명적 오류 | 틀린 출처, 누락된 조항, 깨진 테스트 개수 |
| 재사용성 | 결과를 Markdown, 표, 코드로 바로 옮길 수 있는지 |

예를 들어 웹 조사 결과가 3분 만에 나왔어도 잘못된 링크를 고치는 데 20분이 들었다면 실제 소요 시간은 23분입니다. 반대로 첫 답이 느려도 출처와 비교표가 정확해 검수 시간이 짧다면 업무 도구로 더 나을 수 있습니다.

## 도구를 하나만 구독해야 한다면

지난 2주간 반복한 작업을 열 개만 적고 분류해 보세요.

- 웹 링크를 찾는 일이 절반 이상이면 검색형 도구
- 여러 PDF와 회의록을 비교하는 일이 절반 이상이면 출처 기반 문서 도구
- 코드 읽기, 수정, 테스트가 절반 이상이면 IDE 통합 코딩 도구

도구의 유명세보다 **내가 매주 반복하는 입력과 검증 방식**이 선택 기준입니다. 처음부터 연간 결제하지 말고, 공개 자료와 비민감 작업 하나로 1주일 동안 측정한 뒤 결정하는 편이 좋습니다.
